# Seamless Account Switching - Like Gmail

## Overview

Implemented Gmail-style seamless account switching where users can switch between accounts with just one click, without re-entering passwords.

## How It Works

### Traditional Method (Before)
```
User clicks account → Logout → Redirect to login → Enter password → Login
```

### Seamless Method (Now)
```
User clicks account → Backend generates custom token → Sign in with token → Done!
```

## Architecture

### Frontend Flow
1. User clicks account in dropdown
2. Frontend calls backend `/api/auth/switch-account` with target UID
3. Backend validates and generates custom Firebase token
4. Frontend signs in with custom token
5. Page reloads with new account active

### Backend Flow
1. Receive switch request with target UID
2. Verify current user is authenticated
3. Verify target user exists
4. Generate custom Firebase token for target user
5. Return custom token to frontend

## Implementation Details

### Backend

#### 1. Auth Controller (`authController.ts`)
```typescript
export const switchAccount = async (req: Request, res: Response) => {
  const currentUid = req.user?.uid;
  const { targetUid } = req.body;
  
  // Verify target user exists
  await auth.getUser(targetUid);
  
  // Generate custom token
  const customToken = await auth.createCustomToken(targetUid);
  
  res.json({ success: true, data: { customToken, uid: targetUid } });
};
```

### Frontend

#### 1. Multi-Account Service (`multiAccountService.ts`)
```typescript
export const switchToAccount = async (account: StoredAccount) => {
  // Get current user's token
  const idToken = await currentUser.getIdToken();
  
  // Request custom token from backend
  const response = await fetch('/api/auth/switch-account', {
    method: 'POST',
    headers: { Authorization: `Bearer ${idToken}` },
    body: JSON.stringify({ targetUid: account.uid }),
  });
  
  const { customToken } = await response.json();
  
  // Sign in with custom token
  await signInWithCustomToken(firebaseAuth, customToken);
  
  return true;
};
```

#### 2. Account Switcher (`AccountSwitcher.tsx`)
```typescript
const handleSwitchAccount = async (uid: string) => {
  const account = accounts.find((a) => a.uid === uid);
  
  // Seamless switch using custom token
  await switchToAccount(account);
  
  // Set as active account
  setActiveAccount(uid);
  
  // Reload page with new account
  window.location.href = '/';
};
```

## Security

### Why This Is Secure

1. **Authentication Required**: Current user must be authenticated to request switch
2. **Backend Validation**: Backend verifies both current and target users exist
3. **Custom Tokens**: Firebase custom tokens are short-lived and single-use
4. **No Password Storage**: Never stores or transmits passwords
5. **Audit Trail**: All switches are logged in backend

### Firebase Custom Tokens

- Generated by Firebase Admin SDK (server-side only)
- Short-lived (1 hour by default)
- Single-use (cannot be reused)
- Signed with service account private key
- Can only be generated by backend with admin privileges

## Comparison with Gmail

### Gmail's Approach
- Keeps multiple sessions active simultaneously
- Uses OAuth tokens for each account
- Switches between sessions without re-authentication
- Stores encrypted tokens locally

### Our Approach
- Generates custom tokens on-demand
- Uses Firebase Admin SDK for token generation
- Switches by signing in with custom token
- No local token storage (more secure)

### Similarities
✅ One-click account switching
✅ No password re-entry
✅ Fast and seamless
✅ Multiple accounts supported

### Differences
- Gmail: Multiple simultaneous sessions
- Ours: One active session, seamless switching
- Gmail: Client-side token management
- Ours: Server-side token generation (more secure)

{

## Error Handling

### Frontend Fallback
If seamless switching fails, automatically falls back to traditional method:
```typescript
try {
  await switchToAccount(account);
} catch (error) {
  // Fallback: Traditional method
  await logout(true);
  window.location.href = `/login?switch=${account.email}`;
}
```

### Backend Errors
- **401**: Current user not authenticated
- **400**: Target UID not provided
- **404**: Target user not found
- **500**: Token generation failed
